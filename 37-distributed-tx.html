三十七、分布式事务深化：Seata/TCC/Saga 边界与实战 ​

1. 为什么本地事务的 ACID 在分布式环境下会"失效"？分布式事务的核心难点是什么？ ​

难度：★★

考点：本地 ACID 回顾、CAP 约束、分布式故障模型

参考答案：本地事务靠单库 undo/redo 日志与行锁即可保证 ACID；一旦跨库/跨服务，原子性被网络边界打破。核心难点有三：网络不可靠（消息丢失、超时、乱序）、部分失败（A 成功 B 失败）、脑裂（协调者与参与者失联导致状态不一致）。

深度解析：Seata 官方文档明确指出，分布式事务本质是"在不可靠网络上为多资源访问提供原子性保证"。单库 ACID 的 Isolation 由 MVCC+锁实现，跨库则没有全局锁视图；《Distributed Systems》中强调，P（分区容错）在网络下必选，故只能在 C 与 A 间取舍，这正是 BASE/最终一致的由来。

易错点：误以为"加分布式锁"就等于分布式事务；混淆"网络分区"与"网络延迟"，后者可重试、前者需共识。

追问： ​

追问1：CAP 里为什么 P 几乎不可放弃？ ​

答：真实广域网与微服务调用必然存在丢包、宕机、断连，分区是客观存在的故障而非可关闭的开关；若放弃 P 等于假设永不网络分区，这在生产不可用，故只能在 C/A 间取舍，AP 系统靠最终一致补偿。

追问2：BASE 与 ACID 是否对立？ ​

答：非对立，是适用域不同。ACID 保证单资源强一致，BASE（基本可用、软状态、最终一致）是分布式下对 C 的放宽，通过补偿/重试回到一致，二者在"强一致写"与"跨域最终一致"分层共存。

2. 2PC（XA 协议）的标准流程是怎样的？它的三大致命缺陷是什么？ ​

难度：★★

考点：XA 两阶段提交、同步阻塞、单点、数据不一致

参考答案：阶段一（Prepare）：协调者问各参与者"能否提交"，参与者写 redo/undo 并加锁后回 YES/NO；阶段二（Commit）：全 YES 则发 commit，否则发 rollback。三大缺陷：同步阻塞（资源锁持有至第二阶段）、协调者单点（宕机则参与者阻塞）、数据不一致（协调者发部分 commit 后宕机，部分提交部分未提交）。

深度解析：XA 由 X/Open 定义，依赖资源管理器（RM）与事务管理器（TM）。其阻塞源于"Prepare 后必须等全局决议"，锁长期占用；MySQL XA 在 5.7 后支持，但 Binlog 与引擎日志需内部XA保证一致。Atomic 的 2PC 论文指出，协调者崩溃恢复需持久化决策日志，否则产生"不确定事务"。

易错点：以为 2PC 一定能保证一致；忽略"协调者发 commit 后宕机"这一窗口会造成参与者状态分裂。

追问： ​

追问1：协调者第二阶段刚发完 commit 就宕机，参与者怎么办？ ​

答：已收到 commit 的参与者提交，未收到的会一直阻塞或超时；恢复后协调者查决策日志，若有 commit 记录则通知剩余参与者补提交，无记录则回滚，故必须持久化"全局事务决议"才能避免悬挂。

追问2：为何 XA 在微服务里少用？ ​

答：微服务跨应用难以共享同一 TM，且长事务持锁阻塞、性能差；而且一个慢服务会拖垮全局，不适合高并发互联网场景，多用于同库多表或传统 ESB 强一致短事务。

3. 协调者在 2PC 中宕机，如何用"日志 + 超时"机制收敛到一致状态？ ​

难度：★★★

考点：协调者故障恢复、事务日志、参与者超时推进

参考答案：协调者须先写"全局决议日志"再发 commit；恢复后读日志：有 commit/rollback 则重发；无则按策略决定。参与者在 Prepare 后启用超时，超时未收到决议时，可向其他参与者询问或依据"对方已提交"反推自身应提交（3PC 的思想）。

深度解析：2PC 的改进版 3PC 引入 preCommit 与超时自动提交，缓解阻塞但仍有网络分区下的不一致。Seata 的 TC（Transaction Coordinator）即扮演持久化协调者角色，全局事务状态存于 db/redis，server 重启可重建；官方建议 TC 高可用部署并开启 store.db 持久化，避免决议丢失。

易错点：认为参与者超时就能"自行回滚"永远安全；实际上分区下自行回滚可能与已提交方冲突，需业务补偿而非硬回滚。

追问： ​

追问1：3PC 真能解决 2PC 阻塞吗？ ​

答：只能缓解。3PC 通过 preCommit 让参与者在超时后默认提交，减少阻塞；但网络分区时"默认提交"与"默认回滚"的参与者可能并存，仍可能不一致，因此 3PC 也未大规模取代 2PC。

追问2：Seata 的 TC 高可用怎么落地？ ​

答：TC 作为独立无状态服务，全局事务与锁信息持久化到共享存储（MySQL/Redis），多实例前置 LB；任一实例宕机，其他实例凭持久化状态继续推进，客户端重连即可，决议不丢。

4. TCC 的 Try/Confirm/Cancel 三阶段语义与"资源预留"思想是什么？ ​

难度：★★

考点：TCC 三阶段、业务侵入、预留资源

参考答案：Try 做资源检查与"预留"（如冻结配额而非直接扣减）；Confirm 用预留资源执行业务（真正扣减），需幂等；Cancel 释放预留。与 2PC 不同，TCC 把"准备"做成业务语义的预留，第二阶段无锁竞争、性能高。

深度解析：TCC 出自 Pat Helland《Life beyond Distributed Transactions》。在碳管家场景中，生成报告前先 Try 冻结用户碳配额（quota 表 freeze_amount+），Confirm 时 freeze 转 used；若大模型调用失败则 Cancel 把 freeze 退回。Try 必须"可确认也可取消"，Confirm 只允许成功（不允许再失败）。

易错点：把 Try 写成"直接扣减"——那样 Cancel 无法安全回退，且 Confirm 失败会丢数据；Confirm 必须设计成"只增删预留、不重新校验业务"。

追问： ​

追问1：Confirm 为什么被要求是"必须成功"的操作？ ​

答：因为 Try 已通过校验并预留资源，Confirm 阶段不应再做可能失败的业务判断，只做幂等的资源落定；若 Confirm 仍失败说明是系统故障，应靠重试而非回滚，否则预留资源永久悬挂。

追问2：碳配额冻结与直接扣减如何选型？ ​

答：高并发且需可回退选冻结（TCC）；若业务天然不允许"占用后放弃"则直接扣减+补偿。碳管家报告生成是典型"先占后定"，冻结更稳，避免超发。

5. 什么是 TCC 的"空回滚"？如何用"事务状态记录"识别并安全跳过？ ​

难度：★★★

考点：空回滚、悬挂、状态表

参考答案：空回滚指 Cancel 比 Try 先到达（网络乱序/超时重试），此时 Try 从未执行，Cancel 若直接释放会误操作。解法：在 Try 中先插入"事务状态行"（status=trying）；Cancel 发现无对应 status 或 status≠trying 时，记录"已回滚占位"后直接返回成功，待 Try 到达时因检测到已取消而跳过。

深度解析：Seata TCC 模式要求分支事务注册全局锁与 action 记录；Hmily/TCC-Transaction 框架均维护"事务参与日志表"，Cancel 据此判断是否空回滚并写入"空回滚标记"，防止后续迟到的 Try 真正执行造成资源悬挂。

易错点：在 Cancel 里直接 if(record==null) return; 却不留痕，导致后续 Try 照常执行占用资源——必须写"已取消"标记让 Try 感知。

追问： ​

追问1：空回滚标记如何与防悬挂配合？ ​

答：Cancel 空回时写入 cancel_mark；Try 到达先查：有 cancel_mark 则立即返回（不再预留），实现"防悬挂"；两机制共用同一状态表，顺序无关，靠标记位区分，是 TCC 健壮性标配。

追问2：框架不自动处理时手写要点？ ​

答：用一张 tcc_action 表（xid, action_id, status, create_time），Try 插入、Confirm 更新、Cancel 更新或插入空回标记；所有分支以 xid 为幂等键查表决策，配套唯一索引防并发重复。

6. TCC 的"防悬挂"是什么？Cancel 早于 Try 到达时如何避免资源悬挂？ ​

难度：★★★

考点：防悬挂、状态机、超时时序

参考答案：悬挂指 Cancel 先到并执行了回滚，随后迟到的 Try 又成功预留资源却再无 Cancel，造成资源"永久占用"。防悬挂：Cancel 空回时落"已取消"标记；Try 执行前查标记，若存在则拒绝预留直接返回，使迟到 Try 失效。

深度解析：与第 5 题共用机制但侧重"防止 Try 迟到生效"。Dubbo + TCC 在弱网下常见此问题；阿里 GTS/Seata 的 TCC 接口规范建议 Try 方法首行校验全局事务是否已取消。本质是"以状态表为事实源，拒绝迟到的正向操作"。

易错点：只在 Cancel 做空回保护却忘了在 Try 拦截，结果悬挂依旧；二者必须成对实现。

追问： ​

追问1：如何用唯一索引杜绝重复 Try？ ​

答：tcc_action 表对 (xid, branch_id) 建唯一索引，Try 插入即占位；迟到 Try 因唯一冲突抛错，捕获后查 cancel_mark 决定返回，从而天然防重与防悬挂。

追问2：超时配置对悬挂的影响？ ​

答：Try 超时过长会放大 Cancel 先到的概率；应设合理 Try 超时并配合全局事务超时，让 Cancel 尽早触发空回标记，缩短悬挂窗口。

7. TCC 的 Confirm/Cancel 为什么要幂等？重试策略应该如何设计？ ​

难度：★★

考点：幂等、重试、指数退避

参考答案：Confirm/Cancel 由协调者在网络异常下重复调用，非幂等会导致"重复扣减"或"重复释放"。设计：以 xid+branchId 为幂等键，操作前查状态表，已 Confirm 则直接返回；重试采用指数退避+最大次数，最终交由人工/告警兜底。

深度解析：Seata 在 TC 侧对分支事务做重试，Hmily 用 retry 配置控制次数与间隔。幂等实现常见三态机：init→confirmed/cancelled，状态机驱动保证"同请求多次结果一致"。碳管家配额扣减 Confirm 必须幂等，否则重试造成超扣。

易错点：用"先查后改"非原子实现幂等，并发下仍重复；须用 UPDATE ... WHERE status='init' 受影响行数判定或唯一约束。

追问： ​

追问1：状态机幂等与去重表哪种更好？ ​

答：状态机更适合有生命周期的业务（如配额）；去重表（唯一键+已处理标记）更轻量。二者可结合：去重表防并发、状态机管语义，碳管家推荐"状态机为主、唯一索引兜底"。

追问2：Confirm 重试一直失败怎么办？ ​

答：超过阈值进入"人工订正/告警"，同时记录悬挂预留以便对账；绝不能无限重试拖垮事务，也不应自动回滚（Confirm 不可逆），靠运维补偿修复数据。

8. Saga 长事务的"正向步骤 + 补偿逆向"模型如何设计？ ​

难度：★★

考点：Saga、补偿事务、长流程

参考答案：把长事务拆为有序本地事务 T1…Tn，每步有对应补偿 C1…Cn（逆序执行）。任一 Ti 失败则依次补偿 Cn…Ci+1 回滚。碳管家"报告生成→配额扣减→因子同步→大模型核算"可用 Saga：任一步失败反向补偿前面已成功的步骤。

深度解析：Saga 源自 1987 年 Princeton 论文，面向"长活事务"（hours/days）。与 TCC 区别在它不预留资源，每步直接提交本地事务，靠补偿回退；适合跨多系统、长耗时流程。Seata 提供 Saga 模式（基于状态机 JSON 编排）。

易错点：把补偿写成"反向业务"却未考虑部分成功态；补偿本身也要幂等，且需可重试，否则补偿失败会留半成品。

追问： ​

追问1：Saga 补偿与 TCC Cancel 区别？ ​

答：Cancel 释放"预留"资源、业务未真正发生；Saga 补偿是撤销"已提交"的生效业务（如作废已生成的报告），语义更重、更难完全反转（如已发短信无法撤回）。

追问2：补偿失败如何兜底？ ​

答：补偿需幂等且重试；多次失败转入"告警+人工订正+对账任务"，并标记流程为异常终态，避免脏数据扩散，碳管家用定时对账修复配额偏差。

9. Saga 的"编排（choreography）"与"协调（orchestration）"有何取舍？ ​

难度：★★★

考点：choreography vs orchestration、事件驱动

参考答案：编排（choreography）靠事件驱动：每服务完成即发事件，下游自行触发，去中心、耦合低但流程难观测、易成"事件迷宫"；协调（orchestration）由中心协调器（如 Seata Saga 状态机）按定义依次调用并触发补偿，可控、易观测但中心重。

深度解析：编排典型用 MQ 事件链（报告生成完发 ReportDone，配额服务订阅后扣减）；协调用 Seata Saga 的 *.json 状态机描述节点与补偿。碳管家跨"报告/配额/因子/大模型"四服务，推荐协调式以便统一观测与补偿。

易错点：中小团队盲目用编排导致"无主控、难追踪"；或过度中心化把协调器做成大单体，失去微服务解耦收益。

追问： ​

追问1：如何用事件溯源辅助编排可观测？ ​

答：每步发领域事件并落 event_log，配合 traceId 串联；失败时可由事件回放定位卡点，补偿指令也以事件下发，既保留编排解耦又补回可观性。

追问2：Seata Saga 状态机好处？ ​

答：用声明式 JSON 描述正向/补偿分支，支持重试、跳过、并行；异常自动走补偿路径，集中可视化，降低手写协调代码与补偿遗漏风险。

10. Saga 缺乏隔离性会带来什么问题？如何用"语义锁/交换式补偿"缓解？ ​

难度：★★★

考点：Saga 隔离缺失、脏读、补偿语义

参考答案：Saga 每步提交本地事务，未完成的整体事务对其他事务可见，导致"脏读/丢失更新/模糊读"。缓解：① 语义锁（业务状态机标记"进行中"，禁止外部修改）；② 交换式补偿（补偿不简单回退，而是用逆向业务抵消，如配额多扣用"补回"而非删记录）；③ 关键资源加版本号防并发覆盖。

深度解析：Seata Saga 文档提示 Saga 不保证隔离，需业务层介入。碳管家中"配额已扣减但报告最终失败补偿"，补偿期间若用户并发下单会读到"已扣未定"的中间态；用 status='PROCESSING' 阻止并发扣减可降风险。

易错点：用"物理回滚"实现补偿却忽略并发写；补偿未加版本导致覆盖他人已提交修改。

追问： ​

追问1：语义锁会影响性能吗？ ​

答：会，但仅锁住"进行中"的少量热点资源，比 2PC 全局锁轻；可按资源粒度（用户级而非表级）控制，碳管家按 user_id 加行级状态锁，影响可控。

追问2：交换式补偿举例？ ​

答：报告生成补偿不是删报告表记录，而是写一条 status=CANCELLED 并记审计；配额补偿不是回退原行，而是 INSERT 一条 +freeze 反向流水，保证可追溯且不被并发更新覆盖。

11. 本地消息表如何实现"消息最终一致"？它与 2PC 的本质区别？ ​

难度：★★

考点：本地消息表、最大努力、最终一致

参考答案：在本地事务中同时写业务数据与"消息表（status=pending）"，同库保证原子；另起定时任务扫描 pending 消息发 MQ，成功后置 done。消费方幂等处理。与 2PC 区别：不要求强一致提交，允许短暂不一致、靠重试收敛，无全局锁、性能好。

深度解析：eBay 经典方案。碳管家"配额扣减成功→发'报告可生成'消息"用本地消息表，确保扣减与消息不丢；即使 MQ 暂不可用，消息表落盘后由任务补偿发送。本质是 BASE 而非 ACID。

易错点：消息表与业务不在同一事务（跨库）会破坏原子性；消费方未做幂等导致重复处理。

追问： ​

追问1：消息表扫描压力如何优化？ ​

答：按 status+create_time 建索引分页扫描，配合批量发送与"发送中"状态防并发重复；高吞吐可改用事务消息替代，减少轮询开销。

追问2：本地消息表能保证不丢消息吗？ ​

答：能保证"业务成功则消息必发"（同事务落盘），但不能保证"立即发到"；靠重试最终送达，属最大努力，需消费方幂等兜底，不提供强一致。

12. RocketMQ 事务消息的 half message / commit / rollback 机制如何保证一致？ ​

难度：★★★

考点：RocketMQ 事务消息、两阶段消息、回查

参考答案：生产者发 half message（对消费者不可见）到 Broker；本地事务执行后发 commit（half 转为可消费）或 rollback（丢弃）。若生产者宕机未发终态，Broker 定时回查 Producer 本地事务状态决定提交。碳管家"配额扣减后发 half，报告生成成功再 commit"。

深度解析：RocketMQ 事务消息把"本地事务"与"消息可见性"解耦：half 阶段消息对消费方屏蔽，避免脏消费；回查机制弥补"终态丢失"。其局限是仅保证"发端"一致，消费端仍须幂等。官方强调回查需幂等且快速。

易错点：把 half 当普通消息直接消费；回查逻辑未查真实业务状态而随意返回 UNKNOWN 导致消息堆积。

追问： ​

追问1：回查超时上限与消息堆积？ ​

答：Broker 默认回查 15 次左右后转为"终止/人工"，可配；回查应查库真实状态，避免长期 UNKNOWN 致 half 堆积、下游迟迟不消费。

追问2：与本地消息表比优劣？ ​

答：事务消息省去扫表任务、实时性更好，但强绑 RocketMQ；本地消息表中间件无关、可控性强但需自维护任务。碳管家若已用 RocketMQ 优先事务消息。

13. "最大努力通知"是什么？与 TCC/消息一致在选型上如何取舍？ ​

难度：★★

考点：最大努力通知、弱一致、对账

参考答案：最大努力通知指主动方尽最大次数通知被动方，失败则留对账/人工补救，不保证一定送达（如支付结果通知）。与 TCC（强一致、侵入高）、消息最终一致（保证送达）相比，它一致性最弱但成本最低，适合"结果可查、允许延迟"的场景。

深度解析：碳管家中"因子库更新通知各租户缓存刷新"可用最大努力：发通知+留版本号，消费方按版本拉取，辅以每日全量对账。它不阻塞主流程，是"通知而非事务"。

易错点：把"最大努力"用于资金/配额等强一致场景；未在被动方提供"主动查询"兜底，通知丢失即永久不一致。

追问： ​

追问1：何时必须用 TCC 而非最大努力？ ​

答：涉及"不可超发、不可重复"的资源（配额扣减、库存）必须 TCC 或强一致；最大努力只用于可重算/可对齐的非关键同步（缓存、统计），否则丢通知即资损。

追问2：对账在最大努力里的角色？ ​

答：是对一致性的最后防线：定时比对主动方与被动方状态，差异自动或人工修复；碳管家每日对账因子版本，发现通知遗漏即触发全量补偿同步。

14. Seata AT 模式原理：数据源代理、undo_log、前置/后置镜像与全局锁 ​

难度：★★★

考点：Seata AT、undo_log、镜像、全局锁、回滚

参考答案：AT 在 JDBC 层做代理：执行 SQL 前生成前置镜像（before image），执行后生成后置镜像（after image），与行锁写入 undo_log（本地事务内，保证与业务原子）；注册全局锁防脏写。回滚时按 before image 反向补偿，提交后异步删除 undo_log。

深度解析：Seata 官方 AT 文档描述两阶段：一阶段本地事务提交含业务+undo_log；二阶段 TC 通知提交（删 undo_log）或回滚（按镜像生成反向 SQL）。全局锁（lock_table）防止其他全局事务并发改同一行造成脏写，是 AT 实现隔离的关键。

易错点：undo_log 与业务不在同一库导致回滚失效；未建全局锁索引引发锁表性能问题；AT 仅支持 relational DB 与特定 SQL。

追问： ​

追问1：AT 回滚靠镜像，遇到"中间被改过"怎么办？ ​

答：AT 提交时校验 after image 与当前数据一致（脏写检测），不一致说明被非 Seata 事务改过，抛异常转人工；全局锁正用于阻止这类并发修改，故建议所有写同一资源的服务都接入 Seata。

追问2：AT 与 XA 性能差异来源？ ​

答：XA 在 Prepare 后持锁至全局提交，阻塞久；AT 一阶段即本地提交释放本地锁，仅持全局锁（轻量、内存/表行），整体吞吐更高，是 AT 在互联网流行的原因。

15. Seata AT 与 TCC 该怎么对比选型？侵入性、性能、一致性各有何权衡？ ​

难度：★★

考点：AT vs TCC、侵入性、性能

参考答案：AT 对业务零侵入（只加注解+undo_log 表），一阶段即提交、性能好，但弱隔离、依赖关系型 DB、镜像回滚有局限；TCC 侵入高（手写三阶段）、开发重，但无全局锁竞争、强可控、适合高一致资源。碳管家"报告生成"可用 AT，"配额扣减"宜用 TCC。

深度解析：Seata 文档将 AT 定位为"无侵入自动补偿"，TCC 为"人工精细控制"。AT 适合大多 CRUD 链路；TCC 适合"钱/配额"等不能超发且需预留语义的核心资源。二者可混用：同一全局事务内，AT 管报告、TCC 管配额。

易错点：用 AT 管理配额导致"一阶段已提交、回滚靠镜像"在并发下出现脏写；或全用 TCC 致开发成本爆炸、小场景过度设计。

追问： ​

追问1：能否同一事务混用 AT 与 TCC？ ​

答：可。Seata 全局事务下各分支可分别用 AT/TCC/Saga；TC 统一协调两阶段，AT 分支回滚靠 undo_log，TCC 分支回滚靠 Cancel，最终一致，碳管家正适合此混合。

追问2：AT 不支持的 SQL 怎么办？ ​

答：对批量/复杂 SQL、非关系库改用 TCC 或 Saga 手动补偿；AT 仅覆盖可解析的 SELECT/INSERT/UPDATE/DELETE，超出范围需自定义分支类型。

16. 分布式事务选型决策树：何时选 TCC / Seata AT / Saga / 消息最终一致？ ​

难度：★★★

考点：选型、一致性要求、长事务、性能、侵入性

参考答案：决策维度：① 一致性要求强且资源敏感（配额/库存）→ TCC；② 一般业务链路、低侵入优先 → Seata AT；③ 长活跨多系统、可补偿 → Saga；④ 仅需最终一致、解耦异步 → 消息事务/本地消息表。再结合性能（AT/TCC 优）、开发成本（AT 低）、跨异构（Saga/MQ）取舍。

深度解析：经验法则：强一致+短事务选 TCC/AT；长流程+弱隔离容忍选 Saga；跨团队异步选 MQ。碳管家"配额扣减"TCC、"报告写库"AT、"因子跨服务同步"MQ、"端到端长流程编排"Saga，按子域组合。

易错点：用单一模式解决所有问题；在长事务硬上 2PC 致锁表；在超发敏感场景用 MQ 最终一致却无补偿对账。

追问： ​

追问1：一致性要求"中等"如何定？ ​

答：若业务可接受秒级不一致且能补偿（如缓存、统计）选 MQ；若不一致会导致资损/超发（配额、余额）选 TCC/AT。判断标准是"错误一致态是否产生不可逆损害"。

追问2：性能与一致冲突时优先级？ ​

答：核心资源以一致优先（宁可慢不可错），非核心以性能优先；用读写分离、本地缓存减轻一致方案的性能损耗，碳管家配额路径保一致、报告查询路径保性能。

17. 碳核算"不可重复计算"如何用幂等键+状态机+去重表落地？ ​

难度：★★★

考点：幂等设计、幂等键、去重表、碳核算不可重复

参考答案：以"报告ID+核算维度"为幂等键，写入去重表（唯一索引）或在状态机中标记 CALCULATED；大模型核算前先查：已计算则直接返回旧结果（idempotent=false 排除重试路径），未计算才调用。状态机保证同一报告只进一次 CALCULATED，防止重试/消息重投导致的重复碳排计算与超扣配额。

深度解析：碳管家"调用大模型生成报告"耗钱耗时，且重复计算会重复扣配额。实践：入口层用 Redis SETNX(reportId:calc) 或 DB 唯一约束抢占；配额扣减 Confirm 以 xid 幂等；MQ 消费用 messageId 去重。Seata 全局事务 xid 天然可作幂等键串联全链路。

易错点：只在应用层内存去重，多实例下失效；去重表与业务非原子导致"抢到锁却写库失败"后状态错乱；把 idempotent=false 误用于会真正改变状态的核心扣减。

追问： ​

追问1：idempotent=false 的语义在碳核算里如何正确使用？ ​

答：它表示"本次调用不允许被当作幂等重试而跳过"，用于首次真实计算/扣减；当系统判定为重试（去重表已存在）时应直接返回已存结果而非再算，即"前端可标 idempotent=false，后端据去重表决定是否真正执行"。

追问2：去重表与状态机如何共存？ ​

答：去重表（唯一键）解决并发抢占与消息重投，状态机（init→calculating→calculated→failed）解决生命周期与可观测；前者防重入、后者管流程，碳管家二者叠加，配合 xid 全链路串联与对账兜底。
